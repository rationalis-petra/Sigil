Motivation

The problem of sorting is a relatively simple one, and we have a host of algorithms to sort values: quicksort, bubble sort, insertion sort, and many more. However, sorting is an interesting problem to explore from the perspective of abstraction: an ideal sorting algorithm can take any container of elements and a way of comparing them (e.g. <), then return a new container which contains the elements, but in order. For now, we'll choose a List as our container and Integers as the elements contained within. I'll implement an insertion sort for simplicity: 

(sort : List ℤ → ℤ → List ℤ)
(sort list ≜ ((insert / nil) list where
  (insert : ℤ → List ℤ → List ℤ)
  (insert item list ≜ match list
    (nil → item :: nil)
    (x :: xs → if (item < x)
                  (item :: x :: xs)
                  (x :: insert item xs)))))

This is relatively simple example, but we can do better - we can sort a list containing any type A if we add an extra argument: a function for comparing two values 

(sort : ∀ {A} → (A → A → Bool) → List A → List A)
(sort _≤_ list ≜ ((insert / nil) list where
  (insert : A → List A → List A)
  (insert item list ≜ match list
    (nil → item :: nil)
    (x :: xs → if (item ≤ x)
                  (item :: x :: xs)
                  (x :: insert item xs)))))

Now we can provide different comparison functions to match the type of the list. 

> sort (num.int._<_) [3 1 2]
[1 2 3]
> sort (coll.string._<_) ["cba" "abc" "bca"]
["abc" "bca" "cba"]

We can even sort in descending order by providing the 'opposite' comparison function: 

> sort (num.int._>_) [3 1 2]
[3 2 1]
> sort (coll.string._>_) ["cba" "abc" "bca"]
["cba" "bca" "abc"]

However, it is often the case that there is an one ordering that 'makes sense' for a given type. For numbers, this is the usual '<' relation, for strings, this is lexicographical (dictionary) ordering, and so on. We can implement this idea with a two features: open types and instance arguments, which together form a technique known as typeclasses.

An open inductive type is like a regular inductive type, except that we are able to add constructors at arbitrary points later in the code. 

(induct open (OpenBool : 𝒰))

;; some code
(instance otrue : OpenBool)

;; some more code
(instance ofalse : OpenBool)

This leads to an intersting problem: we are always allowed to produce new values, but we cannot consume them! This is because we may always add new values, so a pattern match cannot be guaranteed to work. 

(to-string : OpenBool → String)
(to-string b ≜ match b 
  (otrue → "true")
  (ofalse → "false"))

;; oh no! we've added a third boolean value! Our to-string function is now
;; incomplete!
(instance oneither : OpenBool)

Despite this, there is a use for open inductive types: they can direct a Seuth search. To demonstrate, let's create a new open type which I'll call Ord. Ord will be indexed by a type and a value: 

(induct open (Ord : (A: 𝒰) → (A → A → Bool) → 𝒰))

We can then introduce some new values: 

(instance int-ord : Ord ℤ (num.int._≤_))
(instance str-ord : Ord String (coll.string._≤_))

So, a question - what does the following Seuth query return? 

seuth ('A : int-ord ℤ 'B)

The only solution that Seuth could produce is:

  • A = int-ord
  • B = num.int._≤_

Inspired by this, we introduce a new argument - an instance argument, which we denote with the special parentheses ⦃ ⦄. The 'type' of an instance argument is like a seuth query with a single unknown parameter, which is denoted with a single tick '. The argument to the function is then the solution which Seuth produces to the query. 

(sort : ∀ {A} → ⦃Ord A '⦄ → List A → List A)
(sort ⦃_≤_⦄ list ≜ ((insert / nil) li≤t where
  (insert : A → List A → List A)
  (insert item list ≜ match list
    (nil → item :: nil)
    (x :: xs → if (item ≤ x)
                  (item :: x :: xs)
                  (x :: insert item xs)))))

This means that we can now do: 

> sort [3 2 4 1]
[1 2 3 4]
> sort ["cba" "abc" "bca"]
["abc" "bca" "cba"]

We can also add extra instances, to extend the range of arguments we can provide to the Seuth function

Overriding a Deduced Instance

You are allowed to provide your own value to the function to override the solution Seuth finds. 

> sort ⦃num.int._≥_⦄ [1 2 3 4]
[4 3 2 1]

Multiple Solutions

There are several pitfalls to be aware of when using this system. The first is that Seuth will produce an error if there are multiple solutions. For example, imagine you did: 

(instance int-ord-2 : Ord ℤ (num.int._≥_))

So now you have two possible solutions to the seuth query: 

seuth ('A : Ord ℤ 'B)

Which are:

  • A = int-ord and B = num.int._≤_
  • A = int-ord-2 and B = num.int._≥_

Because of this, the function-call: 

(sort [1 3 2 4])

Will produce an error!

Deducing instances

Sometimes, one type will depend on another to implement a particular typeclass. Going Back to our example from earlier, can we give an Ord for lists? One solution might be 

(list-≤ : ∀ {A} → (A → A → Bool) → (List A → List A → Bool))
(list-≤ _≤_ l1 l2 ≜ match (l1, l2) 
  (nil, _ → true)
  ((x :: xs), (y :: ys) →
    let (_=_ : A → A → Bool)
        (x = y ≜ x ≤ y ∧ y ≤ x)
     (if (x = y) (list-< _≤_ xs ys) (x ≤ y))))

(instance (list-ord : Ord (List 'A) (list-≤ 'Ex)) ⇐
  (instance (Ord 'A) 'Ex))

A Signatures and Open Types

In the last 

(signature (Monoid [A : 𝒰] : 𝒰)
  (_⋅_ : A → A → A)
  (⍬   : A → A → A))

(open induct (CMonoid : (A : 𝒰) → (Monoid A) → 𝒰))

(structure (str-monoid : Monoid String)
  (s1 ⋅ s2 ≜ coll.string.concat s1 s2)
  (   ⍬    ≜ ""))

(instance (string-monoid : CMonoid String str-monoid))

(structure (int-monoid : Monoid ℤ)
  (n1 ⋅ n2 ≜ num.int._+_ n1 n2)
  (   ⍬    ≜ 0))

(instance (ℤ-monoid : CMonoid ℤ int-monoid))

(msum : ∀ {A} → ⦃CMonoid A⦄ → List A → A) 
(msum ⦃monoid⦄ list ≜ foldl monoid._⋅_ monoid.⍬ list)

> msum [1 2 3 4 5] 
15
> msum ["12" "3" "45"]
"12345"

Don't mention this instance argument and it'll automatically be opened (if it's a structure). 

(msum : ∀ {A} → ⦃CMonoid A⦄ → List A → A) 
(msum list ≜ foldl _⋅_ ⍬ list)

Hierarchy

Let's recall where we're up to so far

  • To make functions like sort more general and useful, we express its' type as "sort sorts an array of any type A so long as you provide it a ≤ operation for that type", or more concretely as sort : ∀ {A} → (A → A → Bool) → Array A → Array A
  • For most types (ℤ, String, etc.), the value of the second argument (the ≤ function) is relatively obvious, and so we'd like to the language to figure it out based on information elsewhere. We do this with open types, and the technique itself is referred to as typeclassing.
  • Finally, we found out that we can use signatures as the deduction-endpoint for our typeclasses.
(signature (Order [A : 𝒰] : 𝒰)
  (= : A → A → Bool)
  (< : A → A → Bool)
  (> : A → A → Bool))
