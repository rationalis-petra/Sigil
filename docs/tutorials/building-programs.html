Programs, and Main

In the previous tutorials, we made use of the REPL to evaluate expressions like 2 + 2 and 3 + 4. However, most code we write is part of a program - code which is in a file somewhere and which, when executed, should do something (e.g. analyze some data, run an app, etc.).

However, the tutorial so far has only covered how to calculate things things - if our program starts, does some kind o big calculation then stops, we won't have any way of knowing what results it got! A program needs some way of taking input from the world, and outputting results into the world. This is IO (Input/Output).

Your First Program

Let's write our first proper program: the (in)famous Hello World. First, create a file called main.gl and add the following code: 

(module main
  (import sys.console))

(main ≜ put-line "Hello, World!")

Finally, run it with glyph -i main.gl. It should print out "Hello, World!".

Let's run through the two parts to this program:

  • The business of the program: (def main (put-line "Hello, World!")). When running the program, Glyph needs to know where it should start. By default, it will look for an action called main, and start there. The thing that main actually does is quite simple: it prints the line "Hello, World!".
  • The Module header is - this is the (module main ...) bit. Module headers can be quite complex, but this one does only two things:
    • It tells Glyph what module name to associate with this file (we're just calling this module main).
    • It tells Glyph what other modules (if any) this module requires. In our case, we need access to 'put-line', which is located in 'sys.console', so we have to import it. Glyph comes with a set of standard, 'built-in' modules, which you can find here.

Actions and The IO Type

So, we're now outputting `Hello, World!', but how does it work?. IO is based around the concept of an action. Unlike a function, which simply calculates a value, an action represents an interaction with the outside world. Actions which perform IO have type IO A, where A is the value an action outputs. For example, the inverse of put-line, get-line has type IO String. This means it does some IO (reads user input) and returns a String. put-line, meanwhile, has a rather odd type: String → IO Unit. In other words, it is a function, which takes a string as input, then returns an action which has return type Unit. This can be a little strange to wrap your head around at first, so I'll rephrase: as put-line has type String → IO Unit the code put-line "hello" represents the action which outputs "hello" and returns a value of type Unit. Unit is somewhat like Bool. If you see a Bool, it could be two values: true or false. By contrast, if you see a Unit, it can be only one value, which is written (). So, if you know an action returns Unit, then you know it must return the value (), and so the result is uninteresting - it contains no information. This is precisely why put-line returns Unit: its' role is to output information, and so it has nothing interesting to tell us.

That's a lot of prose without much code, but I promise we'll get back to that soon. So far, we have learned that IO is performed via actions like put-line "hello" or get-line. To combine simple actions like this into more complex ones, we need is a mechanism for combining actions together into larger actions, to say things like: "First print 'hello, world' then make a beeping sound", or "Repeatedly read in lines from the console until the user types 'exit'". It turns out, we only need three functions to do this: seq, bind and pure.

Sequencing

The first function in our list is seq. Seq represents the idea of "do x and then do y". For example, we can first output "hello, world" then "goodbye, world" with the code: 

(module main
  (import sys.console))

(main ≜
  seq (put-line "hello, world")
      (put-line "goodbye, world!"))

If you want to sequence more than two actions, then seq can be nested as follows: 

(module main
  (import sys.console))

(main ≜
  seq (put-line "hello, world")
      (seq (put-line "dum dee doo")
           (put-line "goodbye, world!")))

This can get somewhat awkward (with many parentheses!) if you want to sequence multiple actions, so >> is offered as an infix alternative. 

(module main
  (import sys.console))

(main ≜
  put-line "hello, world" >> "dum dee doo" >> put-line "goodbye, world!")

The type of Seq should be relatively unsurprising: 

(seq : ∀ {A B} → IO A → IO B → IO B)

Bind

To summarize what we've learned so far:

  • Actions represent things which perform some interaction with the outside world
  • We can combine actions together be sequencing them.

However, what if we want to use the output of one action as the input to another? Imagine that we want to greet our user: first, we get their name, then we say hi: 

Bob         # user input
Hello, Bob! # program output

A first attempt to write this program might look something like: 

(module main
  (import sys.console))

(greet : String → IO String)
(greet name ≜ put-line ("hello, " ⋅ name ⋅ "!"))

(main ≜ greet get-line)

However, this code will not compile: it would return a type error. This is because the "greet" function expects a String argument, but we're giving it an IO String. Oops!

In order to "correctly" write this function, we make a simple adjustment: 

(module main)
(open sys.console)

(greet : String → IO String)
(defn greet [name]
  (put-line ("hello, " ⋅ name ⋅ "!")))

(main ≜ bind get-line greet)

The bind function is quite simple: it takes two inputs:

  • An Action which returns some value of type A
  • A function which takes as input an A and returns an action

Like seq, it will sequence the two actions. However, it will do this by feeding the output from the first action to the input of the function, then running the resultant action.

Bind also has an infix version, >>=, and so programs can be written: 

(module main)
(open sys.console)

(greet : String → IO String)
(greet name ≜ put-line ("hello, " ⋅ name ⋅ "!"))

(main ≜ put-line >>= greet)

Do Notation

Before moving to the final function, pure, I'll introduce a handy bit of notation: do notation. To demonstrate why it's useful, let's first build a relatively simple function: it will read in two lines, then combine them together. 

(main ≜ 
  (get-line >>=
    (λ [line1] (get-line >>=
      (λ [line2] (put-line (line1 ⋅ line2)))))))

If we run it, and type in "hello" <Enter> "bob" <Enter>, then our console would look something like: 

$ glyph -i line-combiner.gl
hello
bob
hellobob
$

While the concept what we're doing is simple, it's getting obscured by the multiple layers of >>= and lambda functions. As we make things more complex, this code will only get more and more unwieldy. With do notation, however, we can re-write this as: 

(main ≜ do
  (line1 ← get-line)
  (line2 ← get-line)
  (put-line (line1 ⋅ line2)))

Much cleaner! In general, do-notation is written as a do followed by a list of expressions which are either written as (var ← expr) or (expr). The last expression must be of the second form, i.e. (expr). As another example, if we wanted to greet someone twice: 

(greet : String → IO Unit)
(greet name ≜ put-line ("hello, " ⋅ name ⋅ "!"))

(main name ≜ do
  (name1 ← get-line)
  (greet name1)
  (name2 ← get-line)
  (greet name2))

Pure

The final function we'll look at is pure. Pure is for when you've got a value of type A, but you want a value of type IO A. As an example, imagine we have my-function : String → Int and we want to create an action which reads in the string from the console, and returns the value from my-function. We could write this as: 

(my-action : IO Int)
(my-action ≜ do
  (val ← get-line)
  (pure (my-function val)))

Or, suppose that we want to read in two lines, but only return the longest one: 

(get-longest-line : IO String)
(get-longest-line ≜ do
  (line-1 ≜ get-line)
  (line-2 ≜ get-line)
  (pure (if (length line1 < length line2) line2 line1)))

Both of these are simple/silly examples, but you'll often find yourself writing code that looks like: 

(my-complex-action : IO <sometype>)
(my-complex-action ≜ do
  (val-1 ← <action>)
  <action>
  (val-2 ← <action>)
  ... 
  (pure <some expression involving val-1 and val-2>))

And so it is handy to keep pure around.
